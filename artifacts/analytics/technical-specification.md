# Техническое задание: мобильный навигатор по достопримечательностям (MVP)

## Описание проекта

Система — мобильное приложение для пеших прогулок по достопримечательностям в выбранном городе или районе. Пользователь задаёт область поиска через поиск населённого пункта по API карт или режим «рядом со мной» на основе геолокации, просматривает каталог точек интереса с фильтрацией по категориям в рамках возможностей API, открывает карточки объектов, собирает кольцевой пеший маршрут по ориентиру длины в километрах со старта от геолокации или точки на карте, получает до трёх вариантов маршрута с нумерацией порядка посещения и суммарной длиной, сохраняет маршруты и объекты в избранное на устройстве и открывает точки во внешнем приложении карт. Пошаговая голосовая навигация внутри приложения в MVP не входит.

## Целевая аудитория

| Сегмент | Цели и контекст | Ограничения и ожидания |
|--------|------------------|-------------------------|
| Турист на 1–2 дня | Быстро получить маршрут нужной длины и понятный порядок точек; при необходимости изучить город через каталог без обязательного маршрута | Мало времени, незнакомый город; желание перейти к «Собрать маршрут» за минимум шагов; использование геолокации «рядом со мной» вместо ввода города в спешке |
| Местный житель / гость с детьми | Предсказуемый километраж, простота, кольцевой маршрут; старт от реального места (парковка, дом) | Семейная прогулка; нужны фильтры по категориям для детского досуга; явные сообщения при недостатке точек для маршрута |
| Деловой путешественник | Минимум шагов: город → список или маршрут → внешние карты; сдержанный интерфейс | Мало времени между делами; нужны загрузка и понятные сообщения при сбоях сети и лимитах API |

## Функциональные требования

### Локация, каталог и фильтрация

1. Система должна отображать экран выбора локации со строкой поиска по API карт при использовании сценария выбора города или области, чтобы пользователь привязал область поиска к населённому пункту.
2. Система должна при вводе допустимого названия и подтверждении выбора из результатов поиска устанавливать область поиска по выбранному элементу и загружать каталог достопримечательностей для этой области, чтобы пользователь видел объекты в нужном месте.
3. Система должна при завершении поиска без подходящих подсказок отображать состояние «ничего не найдено» с возможностью изменить запрос, чтобы пользователь скорректировал строку поиска.
4. Система должна при выборе другого результата поиска после ошибки в названии обновлять каталог под новую область без сохранения предыдущей выборки как текущей, чтобы данные соответствовали выбранной локации.
5. Система должна при повторном открытии экрана выбора локации отображать ранее выбранное название города и выполнять смену города через тот же поиск и список подсказок без отдельного многошагового мастера, чтобы сократить число шагов для делового путешественника.
6. Система должна считать локацию заданной и обеспечивать переход к каталогу после выбора города из подсказок без обязательных полей «район» и «радиус» для этого перехода, чтобы не усложнять выбор области.
7. Система должна при выборе режима «рядом со мной» после выдачи разрешения на геолокацию получать координаты устройства и загружать каталог POI в круге с центром в этой точке и фиксированным радиусом поиска из настроек клиента (одно значение для всех пользователей в сборке), чтобы пользователь не вводил название города.
8. Система должна при отказе в геолокации в режиме «рядом со мной» показывать объяснение и предлагать выбор города вручную или повторный запрос разрешения, чтобы пользователь имел альтернативный путь.
9. Система должна при отображении каталога для выбранной локации и наличии категорий в ответе API позволять пользователю выбрать одну или несколько доступных категорий и обновлять список так, чтобы в нём остались только объекты, соответствующие выбранным фильтрам, либо отображался пустой список с пояснением, чтобы сузить подборку.
10. Система должна при сбросе активных фильтров обновлять список и показывать все объекты текущей локации без ограничения по категориям на той же странице пагинации, а при пагинации возвращать на первую страницу, чтобы восстановить полную выдачу.
11. Система должна при доступном выборе нескольких категорий оставлять в списке объекты, принадлежащие хотя бы одной из выбранных категорий (логика «ИЛИ»), чтобы объединять категории в одной выдаче.
12. Система должна при режиме только одного активного фильтра категории при выборе другой категории полностью заменять список объектами только этой категории, чтобы соответствовать ограниченному режиму интерфейса.
13. Система должна позволять просматривать каталог и открывать карточки без обязательного запуска сценария «Собрать маршрут» и без блокировки выхода из сценария каталога, чтобы пользователь изучал город в своём темпе.
14. Система должна при сворачивании приложения с экрана каталога или карточки из каталога и повторном открытии восстанавливать экран, на котором пользователь был до сворачивания, без принудительного перехода на экран «Собрать маршрут», чтобы сохранить контекст просмотра.

### Карточка достопримечательности

15. Система должна при выборе объекта из списка открывать карточку с отображением как минимум названия и координат, а также описания и медиа, если они присутствуют в ответе API, чтобы пользователь оценил район и объект.
16. Система должна при возврате «назад» с карточки объекта из каталога отображать предыдущий экран списка с той же выбранной локацией и теми же активными фильтрами, что до открытия карточки, чтобы сохранить контекст фильтрации.
17. Система должна при наличии в ответе API названия и координат отображать название и координаты (широта и долгота или одна строка координат) в первой видимой области карточки до первой прокрутки содержимого, чтобы решение принималось за один экран.
18. Система должна при наличии в ответе API описания и медиа отображать их в соответствии с данными API, а при отсутствии полей показывать аккуратное пустое состояние без «битых» блоков, чтобы интерфейс оставался целостным.

### Построение кольцевого маршрута и варианты

19. Система должна при нажатии «Собрать маршрут» на экране списка достопримечательностей сразу открывать экран ввода ориентира длины и выбора старта без обязательных промежуточных шагов, кроме системных запросов разрешений геолокации, чтобы сократить путь к построению маршрута.
20. Система должна при уже заданной локации переносить контекст области поиска на экран маршрута без повторного ввода города, чтобы не дублировать ввод.
21. Система должна при разрешённой геолокации и полученных координатах на экране построения маршрута принимать ввод ориентира длины в диапазоне 2–50 км и подтверждение старта «текущее местоположение», чтобы использовать координаты как старт кольца и ориентир длины для расчёта.
22. Система должна при вводе длины вне диапазона 2–50 км отображать валидацию с указанием допустимых границ и не запускать построение маршрута, чтобы исключить недопустимые параметры расчёта.
23. Система должна при выборе старта «точка на карте» и установке маркера в допустимой области использовать координаты маркера как старт кольцевого маршрута при расчёте, чтобы начать маршрут от выбранного места на земле.
24. Система должна при выборе старта «текущее местоположение» при включённой геолокации использовать актуальные координаты устройства как старт, а при недоступности GPS показывать сообщение и предлагать выбор точки на карте, чтобы пользователь задал старт явно.
25. Система должна при наличии в текущей выборке для маршрута не менее трёх достопримечательностей и успешном запуске построения отображать от одного до трёх вариантов маршрута, где у каждого варианта указаны суммарная длина в километрах или метрах и порядок посещения 1…N при N ≥ 3, маршрут кольцевой (старт и финиш совпадают), а суммарная длина включает замыкающий отрезок к старту, чтобы пользователь выбрал подходящую нагрузку.
26. Система должна при доступном после отбора объектов для маршрута менее чем трёх достопримечательностях при запуске построения не показывать успешный список вариантов и отображать сообщение о необходимости не менее трёх точек с предложением расширить область или ослабить фильтры, чтобы пользователь скорректировал условия.
27. Система должна при отображении нескольких вариантов маршрута показывать у каждого варианта собственную итоговую длину, видимую без перехода в скрытые детали, чтобы сравнение выполнялось на одном уровне.
28. Система должна при нескольких вариантах на экране результатов обеспечивать соответствие нумерации остановок и длины выбранному варианту без смешения данных разных вариантов, чтобы отображение оставалось однозначным.
29. Система должна при предыдущей попытке построения с сообщением об ошибке или недостатке данных после изменения длины, фильтра, области или старта и повторном нажатии «построить» выполнять новый запрос расчёта и не отображать одновременно карточки успешного маршрута от старой попытки и сообщение об ошибке новой, чтобы на экране было актуальное состояние последней попытки.
30. Система должна при успешном новом расчёте отображать только данные актуальной попытки на экране результатов, чтобы исключить путаницу между попытками.

### Избранное

31. Система должна при просмотре экрана с выбранным вариантом маршрута и нажатии «сохранить в избранное» сохранять в разделе избранного суммарную длину варианта и упорядоченный список остановок с координатами и отображать запись в списке избранного, чтобы пользователь вернулся к маршруту позже.
32. Система должна сохранять данные маршрута в локальном хранилище устройства (суммарная длина, порядок остановок, координаты каждой остановки) при сохранении в избранное, чтобы запись воспроизводилась на том же устройстве.
33. Система должна при повторном открытии избранного на том же устройстве для ранее сохранённого маршрута отображать те же длину, порядок точек и координаты, что были сохранены, чтобы данные не искажались между сеансами.
34. Система должна при наличии в избранном хотя бы одного маршрута или объекта при открытии раздела избранного показывать список с различением типа «маршрут» / «объект» и возможностью открыть элемент за один шаг от списка, чтобы обеспечить быстрый доступ.
35. Система должна при пустом избранном отображать пустое состояние с подсказкой, как добавить маршрут или объект, чтобы пользователь понимал следующий шаг.

### Внешние карты и навигация вне приложения

36. Система должна при открытой карточке объекта с валидными координатами и нажатии «открыть в картах» запускать внешнее приложение карт с передачей широты и долготы через Intent с `geo:` на Android или открытие карт по координатам на iOS, чтобы пользователь увидел точку во внешнем навигаторе.
37. Система должна при реализации с встроенным фрагментом карты вместо внешнего приложения отображать на фрагменте маркер в координатах объекта из API, чтобы выполнить сценарий просмотра на карте внутри приложения.
38. Система должна при невозможности открыть внешние карты отображать сообщение об ошибке и дублировать координаты объекта текстом с кнопкой «скопировать координаты», чтобы пользователь сохранил позицию альтернативным способом.
39. Система должна при просмотре списка остановок маршрута и нажатии «открыть в картах» у конкретной остановки передавать во внешнее приложение карт широту и долготу этой остановки через механизм ОС (Android `geo:`, iOS — системная ссылка с координатами), чтобы навигация шла к выбранной точке.
40. Система должна при выборе «открыть в картах» для всего маршрута на экране варианта открывать внешнее приложение карт с координатами первой остановки выбранного варианта, оставляя остальные точки доступными по отдельным действиям на том же экране, чтобы сохранить поэтапный выход во внешний навигатор.
41. Система должна при сбое открытия внешнего приложения карт по вине ОС показывать сообщение об ошибке и не отображать встроенную пошаговую навигацию в приложении, чтобы не подменять заявленный объём MVP.
42. Система должна при открытой карточке из списка и нажатии «открыть в картах» выполнять переход во внешнее приложение карт без обязательного посещения экрана «собрать маршрут», чтобы сократить путь к навигации.
43. Система должна при возврате назад с карточки, открытой из списка, снова показывать каталог без автоматического запуска сценария маршрута, чтобы не навязывать построение маршрута.
44. Система должна при отсутствии или некорректности координат точки не выполнять открытие карт и показывать сообщение об ошибке данных, чтобы не передавать невалидные параметры во внешнее приложение.

### Загрузка данных и сообщения о сбоях API

45. Система должна при выполнении запроса каталога или построения маршрута до получения ответа API отображать индикатор загрузки (анимированный индикатор или скелетон списка) в области появления контента, чтобы пользователь видел, что приложение не зависло.
46. Система должна при завершении запроса успешно или с ошибкой скрывать индикатор загрузки и показывать контент или сообщение об ошибке, чтобы интерфейс отражал итог запроса.
47. Система должна при отсутствии сети или завершении запроса по таймауту при загрузке каталога или построении маршрута отображать текст о проблеме сети и кнопку «повторить», по нажатию повторять тот же запрос, чтобы пользователь восстановил загрузку без угадывания причины.
48. Система должна при ответе API с кодом ограничения частоты запросов (например HTTP 429) или сообщением о превышении квоты при запросе к данным карт отображать понятное сообщение о временной недоступности сервиса без отображения кодов ошибок в тексте для пользователя и предоставлять кнопку «повторить», повторяющую запрос после паузы, инициированной пользователем, чтобы отличить лимит от пустой выдачи по региону.

## Нефункциональные требования

### Локация, каталог и фильтрация

1. Система должна завершать отображение индикатора загрузки каталога не позднее чем в момент получения ответа API, ошибки или таймаута, зафиксированного для запроса, чтобы отсутствие ответа не оставляло индикатор бесконечно видимым при завершённом запросе.
2. Система должна использовать для режима «рядом со мной» один фиксированный радиус поиска POI в метрах или километрах, заданный в настройках клиента для сборки и одинаковый для всех пользователей этой сборки, чтобы область выборки была воспроизводимой при приёмочных проверках.

### Карточка достопримечательности

3. Система должна отображать в карточке координаты с точностью не ниже, чем предоставляет API (без сокращения до потери однозначного открытия в картах), чтобы внешние карты получали согласованные с данными источника значения.

### Построение кольцевого маршрута и варианты

4. Система должна ограничивать ввод ориентира длины маршрута значениями от 2 до 50 км включительно на границе ввода до отправки расчёта, чтобы 100% недопустимых по диапазону значений блокировались до вызова API расчёта.
5. Система должна формировать не более трёх вариантов маршрута на один успешный расчёт при достаточном числе точек в выборке, чтобы число вариантов не превышало заявленное в MVP.

### Избранное

6. Система должна обеспечивать сохранение записи избранного маршрута в локальном хранилище так, чтобы после перезапуска приложения на том же устройстве читаемость сохранённых полей (длина, порядок, координаты) составляла 100% для записей, не удалённых пользователем, чтобы данные не терялись между сеансами при штатной работе хранилища.

### Внешние карты и навигация вне приложения

7. Система должна формировать для передачи во внешние карты координаты в виде числовых значений широты и долготы, соответствующих карточке или остановке маршрута в ответе API или сохранённому маршруту, чтобы приёмочная проверка «координаты на входе внешних карт совпадают с источником» давала положительный результат при валидных данных.

### Загрузка данных и сообщения о сбоях API

8. Система должна показывать индикатор загрузки во всех запросах каталога и построения маршрута, пока ответ не получен и не обработан, чтобы в каждом таком запросе было проверяемое визуальное состояние ожидания.
9. Система должна для ответов с HTTP 429 и аналогичными сигналами квоты не включать в пользовательский текст сырой код ответа HTTP, чтобы сообщение содержало ноль встроенных технических кодов в строке для пользователя.

### Общие нефункциональные требования по категориям

Ниже зафиксированы требования в терминах классификации из `nfr-non-functional-requirements.md` (производительность, надёжность, доступность, безопасность, юзабилити, поддерживаемость, масштабируемость). Нумерация сквозная по разделу «Нефункциональные требования».

#### Производительность (Performance)

10. Система должна выполнять сетевые запросы каталога и построения маршрута асинхронно, не блокируя отрисовку уже показанного UI дольше, чем допускает целевая платформа для отзывчивости интерфейса, а ожидание ответа сопровождать индикатором загрузки согласно п. 45–46 и NFR 8.
11. Система должна при пагинации каталога позволять просматривать уже загруженные элементы списка без блокировки прокрутки до завершения запроса следующей страницы, если платформа и реализация это поддерживают.

#### Надёжность (Reliability)

12. Система должна при обрыве связи или ошибке запроса при загрузке каталога или расчёте маршрута сохранять уже отображённые пользователю данные (список, фильтры, выбранную локацию) и не подменять их пустым экраном без сообщения об ошибке, согласно таблице ошибок (в т.ч. смена сети Wi‑Fi ↔ мобильная).
13. Система должна при сбое записи нового элемента в избранное не затирать и не портить ранее сохранённые записи в локальном хранилище.

#### Доступность (Availability)

14. Функции, зависящие от внешних API карт и POI, считаются доступными пользователю только при доступности соответствующих сервисов; при недоступности или ответах 5xx клиент отображает понятное сообщение о временной недоступности и действие «повторить», не выдавая это за отсутствие объектов в регионе.

#### Безопасность (Security)

15. Система не должна выводить в пользовательские сообщения об ошибках и в экраны, доступные скриншоту, секреты сборки (ключи API, токены); при диагностическом логировании в релизной сборке не записывать в логи непосредственно ключи и полные токены.
16. Данные избранного и история выбора локации хранятся локально на устройстве; передача координат во внешние карты выполняется только через механизмы ОС и с валидированными значениями согласно функциональным требованиям и таблице ошибок.

#### Юзабилити (Usability)

17. Сообщения об ошибках и пустых состояниях формулируются понятным пользователю языком без сырых кодов HTTP и без внутренних идентификаторов API в тексте на экране (согласовано с NFR 9 и таблицей ошибок для 429/5xx/SSL).
18. Ключевые сценарии сокращения числа шагов (выбор локации, переход к каталогу и маршруту, внешние карты без лишних экранов) соответствуют функциональным требованиям разделов «Локация, каталог и фильтрация», «Построение кольцевого маршрута» и «Внешние карты».

#### Поддерживаемость (Maintainability)

19. Радиус поиска в режиме «рядом со мной», таймауты HTTP-запросов и базовые параметры подключения к API задаются конфигурацией клиента или сборки в одном месте, без дублирования «магических чисел» по экранам, согласованно с NFR 2 и функциональным требованием о фиксированном радиусе.
20. Поля карточки POI и набор категорий для фильтрации должны по возможности опираться на ответ API и схему данных, допуская расширение списка полей без изменения пользовательских сценариев, если API сохраняет обратную совместимость.

#### Масштабируемость (Scalability)

21. Каталог в рамках одной локации загружается порциями при наличии пагинации в API; клиент не обязан загружать весь набор POI региона одним запросом, если API предусматривает постраничную выдачу.
22. Список избранного отображается как прокручиваемый; при введении в реализации лимита числа записей применяется поведение из таблицы ошибок («Избранное: лимит записей»).

## Описание обработки ошибок

| Ситуация | Ожидаемое поведение системы |
|----------|----------------------------|
| Поиск города: пустая строка, только пробелы или строка короче минимальной длины | Не выполнять лишние запросы к API; при необходимости показать правило ввода у поля (минимальная длина задаётся в реализации согласованно с API). |
| Поиск города: сеть недоступна или таймаут | Сообщение о проблеме сети; кнопка «повторить» повторяет тот же запрос; не смешивать с состоянием «город не найден» в базе. |
| Поиск города: успешный ответ без подходящих подсказок | Состояние «ничего не найдено» с возможностью изменить запрос. |
| Фильтры: выдача пуста при активных фильтрах | Текст о пустой выдаче по фильтрам; доступен сброс фильтров; не показывать немой нулевой экран без пояснения. |
| Фильтры: противоречивая комбинация (если применимо в UI) | Предупреждение или баннер с предложением сбросить фильтры до запроса или после пустого результата — без «тихого» нуля. |
| Ввод длины маршрута: не число, несколько разделителей, пробелы, вне 2–50 км, пустое поле | Блокировка отправки расчёта; inline-сообщения после blur/submit; для диапазона — указание границ «от 2 до 50 км»; для формата — сообщение о вводе одного числа; нормализация пробелов по правилам ввода. |
| Геолокация: отказ в разрешении для «рядом со мной» | Объяснение и выбор города вручную или повторный запрос разрешения. |
| Геолокация: недоступен GPS при старте «текущее местоположение» | Сообщение; предложение выбрать точку на карте. |
| Геолокация: подозрительные координаты (null, 0;0 и т.п.) | Не строить маршрут без явного подтверждения; предложить ручной выбор точки (согласно negative-cases). |
| Каталог региона: пустой список POI | Empty state с предложением расширить область, сменить город или ослабить фильтры; отличать от сетевой ошибки. |
| Построение: в выборке меньше трёх достопримечательностей | Не показывать успешные варианты; сообщение о минимуме трёх точек; рекомендации расширить область или снять фильтр. |
| Построение: сеть недоступна или таймаут | Сообщение о сети; «повторить»; не показывать пустой список как «нет объектов». |
| API: HTTP 5xx | Сообщение о временной недоступности сервиса без обвинения пользователя; «повторить». |
| API: HTTP 429 / квота | Понятное сообщение о лимите без кодов в тексте для пользователя; «повторить» после паузы пользователя; не интерпретировать как «в городе пусто». |
| SSL / сертификат | Краткий текст без технического дампа; рекомендация проверить дату, сеть, VPN. |
| Частичная загрузка списка | Не маскировать обрыв полным списком; явное состояние ошибки блока или догрузка по правилам реализации. |
| Внешние карты: нет приложения или отклонён запрос ОС | Сообщение об ошибке; координаты текстом; кнопка «скопировать координаты». |
| Внешние карты: координаты отсутствуют или некорректны | Действие не выполняется; сообщение об ошибке данных. |
| Избранное: ошибка записи в локальное хранилище | Сообщение пользователю; повтор сохранения; не затирать предыдущие записи при сбое новой (согласно negative-cases). |
| Избранное: дубликат маршрута | Уведомление «уже в избранном» без дублирования записи. |
| Избранное: лимит записей (если введён в реализации) | Сообщение о достижении лимита; указание, что удалить или упростить. |
| Смена сети Wi‑Fi ↔ мобильная во время запроса | Повтор или отмена без краша; сохранение уже показанных данных без необоснованного сброса UI. |
| Облако / авторизация (вне основного MVP локального избранного) | Для будущих экранов синхронизации: редирект или замок «войти»; 401 не маскировать под общую сетевую ошибку; чужие id — экран «не найдено» / «нет доступа». |
