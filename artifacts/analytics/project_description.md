# Описание проекта: мобильный навигатор по достопримечательностям

## Позиционирование

Рабочее название — мобильное приложение для пеших прогулок по достопримечательностям в выбранном городе или окрестности текущего места. Пользователь задаёт область поиска: населённый пункт через подсказки по API карт либо режим «рядом со мной» (каталог POI в круге вокруг геолокации; радиус — фиксированное значение из настроек клиента в сборке), получает подборку мест из данных популярных картографических сервисов, указывает желаемую длину прогулки в километрах и получает готовый маршрут с порядком посещения (нумерация 1…N). По умолчанию маршрут строится кольцом: старт и финиш совпадают (старт кольца — текущая геолокация или точка на карте, см. ТЗ).

*Пример для иллюстрации сценария (не ограничение продукта):* Коломна или любой другой город — приложение должно одинаково поддерживать локации, для которых выбранный провайдер карт возвращает релевантные объекты.

---

## Проблема и ценность

Туристы, жители и деловые путешественники часто тратят время на сбор информации из разных источников и непонятно, в каком порядке обходить точки и какую дистанцию получится набрать. Приложение снимает эту нагрузку: привязка к нужному месту через API карт, один ввод длины — несколько осмысленных вариантов маршрута с понятной последовательностью и оценкой длины в метрической системе; для занятых пользователей важны короткий путь к маршруту и предсказуемое поведение при сбоях сети и лимитах API.

## MVP: ключевые функции

1. Каталог достопримечательностей в выбранной локации — пользователь задаёт область так: поиск населённого пункта по API карт (выбор из подсказок, без обязательных полей «район» и «радиус» для перехода к каталогу) или режим «рядом со мной» (загрузка POI в круге с центром в точке геолокации и фиксированным радиусом из настроек клиента). Список и фильтрация по категориям в рамках ответа API (мультивыбор, логика «ИЛИ» и сброс — по ТЗ); переход к карточке объекта. Данные объектов подгружаются через API популярных сервисов карт (например, экосистемы Google Maps Platform, Яндекс.Карты API, Mapbox, данные на базе OpenStreetMap и аналоги — конкретный набор фиксируется в техническом задании).
2. Карточка достопримечательности — сведения из ответа API (название, описание при наличии, фото/ссылки, координаты), открытие точки во внешнем приложении карт или встроенный фрагмент карты — по решению ТЗ; пошаговая голосовая навигация вне MVP.
3. Построение пешего маршрута (маршрутизация) — ввод ориентировочной длины в км; режим кольцо по умолчанию (старт = финиш: текущее местоположение или точка на карте); выдача до трёх вариантов маршрута с нумерацией порядка посещения и указанием итоговой суммарной длины каждого варианта.
4. Избранное — сохранение объектов и/или сохранённого маршрута для быстрого возврата (без социальных функций в MVP).

---

## Ключевой сценарий

1. Пользователь открывает приложение и задаёт локацию для каталога: поиск города по подсказкам API или «рядом со мной» (после разрешения геолокации — выборка POI в круге с заданным в клиенте радиусом).
2. Просматривает каталог достопримечательностей, при необходимости фильтрует по категориям (в рамках ТЗ и API).
3. Переходит в режим «Собрать маршрут»: указывает ориентир длины, например 10 км, подтверждает стартовую точку (геолокация или выбор на карте).
4. Получает до трёх карточек вариантов («короче», «около», «длиннее» ориентира — см. раздел решений) с порядком 1…N и длиной каждого варианта.
5. При желании сохраняет объект или выбранный маршрут в избранное.

## Принятые продуктовые решения и допущения


| Тема              | Решение                                                                                                                                                                                                                                                                                           |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Платформа         | Мобильный клиент (целевой образ — нативное приложение iOS/Android; в рамках курса допускается веб-MVP с теми же пользовательскими сценариями, например адаптивный интерфейс в браузере).                                                                                                          |
| География         | Любой город/регион, где выбранный картографический API позволяет получить выборку POI и координаты; границы и качество данных определяются провайдером и квотами.                                                                                                                                 |
| Источник данных   | API популярных сервисов карт (геокодирование, поиск мест, категории достопримечательностей, при необходимости — матрица расстояний или пешеходные сегменты, если провайдер это даёт). Дублирование полного справочника в собственной БД в MVP не требуется; кэширование ответов — по ТЗ.          |
| Единицы           | Метрическая система; пользователь вводит километры; внутренняя логика может оперировать метрами.                                                                                                                                                                                                  |
| Длина маршрута    | Оценивается как сумма расстояний между точками в порядке посещения; для кольца включается возврат к старту. Время не используется как целевая метрика (скорость у всех разная).                                                                                                                   |
| Модель расстояния | Базовый вариант: по прямой между координатами (например, гаверсинус). Усиление: где доступно по API провайдера — пешеходная длина между остановками (например, Directions/Matrix API) с fallback на прямую; детали — в ТЗ.                                                                        |
| Варианты маршрута | Пусть L — ориентир в метрах. Короче: D \in [0{,}8L, 0{,}95L] (fallback — ближайший допустимый «короче»). Около: D \in [0{,}92L, 1{,}08L] (fallback — минимум |D-L|). Длиннее: D \in [L, L + \min(2000\text{ м}, 0{,}20L)] (при отсутствии — смягчённый поиск или скрытие категории с пояснением). |
| Ввод длины        | Ориентир L: минимум 2 км, максимум 50 км (MVP).                                                                                                                                                                                                                                                   |
| Минимум остановок | Не менее трёх достопримечательностей в предлагаемом маршруте (если в выборке API недостаточно точек — явное сообщение и не «пустой» успех).                                                                                                                                                       |

## Вне MVP и риски

- Учёт рельефа, пробок, велосипеда vs пеший — вне MVP (заявлен пеший сценарий).
- Глобальная оптимизация «как TSP» — не требуется; достаточно эвристического подбора и прозрачных правил fallback.
- Зависимость от API: лимиты запросов, изменение форматов, разная полнота POI по городам — учитываются в ТЗ (обработка ошибок сети, пустых результатов, превышения квот).
